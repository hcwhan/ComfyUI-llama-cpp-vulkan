## Vulkan / 设备管理

- 选卡语义: 显式选卡传 `SPLIT_MODE_NONE + main_gpu 索引` 与上游 `llama_prepare_model_devices`
  的 "NONE 时以 main_gpu 为 model->devices 下标取单卡" 逻辑吻合 (对照上游源码确认).
- 核显收集规则: llama.cpp 上游的 IGPU 分支只收第一个核显 (`if (igpus.empty())` 才 push),
  且仅在无独显时加入. 已对照本地 JamePeng 仓库 vendored llama.cpp
  (`ComfyUI-X/repos/llama-cpp-python_JamePeng/vendor/llama.cpp`) 直查 `src/llama.cpp`
  的设备收集与 `LLAMA_SPLIT_MODE_NONE` 取卡逻辑确认, 与 `_selectable_devices()` 假设吻合.
- ggml Vulkan 设备的 `ggml_backend_dev_name` 返回 "Vulkan0/Vulkan1..." 形态的唯一
  编号名, `DEVICE_LABEL_TEMPLATE` 含 {name}, 双同型号显卡不会产生重复的下拉 label,
  显式选卡的 label 反查不受影响.

## Chat handler / thinking

- `_HANDLER_SPECS` 对 wheel `llama_multimodal` 的覆盖完整 (全部 24 个 handler 类均被引用,
  含借给 DeepSeek-OCR 的基类 MTMDChatHandler).
- Gemma4 的思考块格式是 `<|channel>thought ... <channel|>` 而非 `<think>`;
  enable_thinking 参数只影响模板渲染: True 注入 `<|think|>` 系统块,
  False 向 generation prompt 注入预闭合空思考通道.
- Gemma4 E2B/E4B 实测结论 (各自 Q4_K_M + BF16 mmproj, Vulkan 后端,
  文本/图像各两组用例, 两型号行为一致):
  向 E2B/E4B 传 enable_thinking=False 安全 (不崩溃, 正常生成并正确收尾);
  wheel 所称 E 系列开关 "不受支持" 的实际表现 (仅文档级说明, 无运行时拒绝分支):
  False 时模型仍以纯文本思考, 并自行输出无开标签的 `<channel|>` 分隔符;
  True 时输出完整 `<|channel>thought ... <channel|>` 结构. 两种情况按
  "取最后一个 `<channel|>` 之后" 均可清理干净.
  视觉路径正常 (纯色图正确识别), BF16 mmproj 与 wheel 推荐一致.
- GLM-4.1V 的输出形态为 `<think>...</think><answer>正文</answer>` (官方推理代码按
  `<answer>(.*?)</answer>` 提取正文); wheel handler 以 `</answer>` 为 stop token,
  闭合标签不进入文本, 因此剥掉 think 块后正文以 `<answer>` 开头.
  官方推理代码与模型文档的核实依据已足够, 无需本地 GGUF 实测.
  GLM-4.6V 不受影响 (stop 列表无 `</answer>`, 模板不用 answer 包裹).

## ComfyUI 侧机制

- 结果缓存: outputs 缓存签名 = 本节点与全部祖先各自的 immediate signature 依序拼接,
  每份 = 类名 + IS_CHANGED 结果 + 排序后的输入项 (连线输入编码为
  `(ANCESTOR, 祖先序号, 输出口)` 引用, 非连线输入取常量原值);
  `IS_CHANGED` 在每次排队开始时统一求值, 且刻意只传常量 (连线输入不解析),
  实现须用 `**kwargs` 兜底签名. `NOT_IDEMPOTENT` 只把 node_id 掺入本节点签名
  (解决同图同参节点去重), 不影响跨排队缓存.
- Unload Model 节点维持缓存语义 (上游输出未变化时不重复卸载) 是有意决策,
  已在 docstring/tooltip 说明; 若未来要强制执行, 最优解是条件 `IS_CHANGED`
  (`LLAMA_CPP_STORAGE.llm is not None` 时返回 NaN, 否则稳定值),
  既修复跨工作流场景又不产生常态化下游重算.
- `InterruptProcessingException` 为 BaseException 子类 (查 ComfyUI 源码确认),
  插件 `except Exception` 不会误吞中断; InterruptWatcher 持续重复 set 的抗竞态设计
  与 wheel 每次请求开始 `_abort_event.clear()` 匹配.
- 中断标志无泄漏风险: ComfyUI 在每次 prompt 开始执行时统一复位中断标志
  (execution.py `execute_async` 开头 `nodes.interrupt_processing(False)`),
  进度 hook 的 `throw_exception_if_processing_interrupted` 抛出时也自行清 flag;
  插件各处直接 `raise InterruptProcessingException` 而不清 flag, 不会把中断
  泄漏到下一次排队.
- 列表语义声明组合 (`json_to_bboxes` INPUT_IS_LIST+OUTPUT_IS_LIST, `bboxes_to_bbox`
  INPUT_IS_LIST, `split_instruct_output` OUTPUT_IS_LIST 等) 正确, widget `[0]` 解包无遗漏.
  列表 map 执行对长度不齐的输入按 "重复末元素" 广播 (execution.py `slice_dict`):
  `bboxes_to_segs` / `bboxes_to_mask` 不声明 INPUT_IS_LIST, 接收多组 BBOX +
  单张 IMAGE 时逐组执行并复用该图, 输出 SEGS/MASK 列表, 接法成立.
- `ProgressBar.update_absolute` 有节流 (间隔 >= 100ms 且变化 >= 0.5% 才推送), 但首次
  (`_last_sent_value < 0`), 末次 (value >= total) 与带 preview 的推送恒即时发送 -
  cqdm "构造即推 0/N" 与逐张 `update(1)` 的末帧推送均不受节流影响.
- 前端 Free 按钮 (POST /free) 走 prompt 队列 flag, 由执行线程在任务间隙处理,
  `patched_unload_all_models` 不存在与推理并发清理模型的竞态.

## 下游生态兼容

- Impact Pack 的 SEG bbox dtype: 其生态内 float32 ndarray 与整型 tuple 两种形态并存
  (最主要的 SEGS 生产者 UltraBBoxDetector 把 YOLO xyxy 的 float32 ndarray 原样透传,
  `mask_to_segs` 则产整型 tuple), 下游节点本须兼容 float;
  本插件 `bboxes_to_segs` 写 float32 与检测器路径的惯例一致, 无需改整型.

## llama-cpp-python wheel 契约

- `llama_cpp_parameters` 12 个字段中 11 个字段名与 `create_chat_completion` 签名直接对应
  (含 JamePeng 分支特有的 `present_penalty` 拼写), UI 键 `max_gen_tokens` 由 instruct 的
  `_run()` 映射回库参数 `max_tokens`; `max_tokens=0` = 不限制与库实现一致.
- hybrid 重置路径 (`n_tokens = 0` + `_ctx.memory_clear(True)` + `_hybrid_cache_mgr.clear()`)
  与 wheel 内部清理写法一致; 刻意用 `_model.is_hybrid()` C API 而非公开 `llm.is_hybrid`
  (含 SWA, 语义更宽) 是正确选择.
- `vram_limit=-1` 透传 `n_gpu_layers=-1` 的 auto 语义与 wheel `_parse_n_gpu_layers` 一致
  (源码确认: "auto" -> -1, "all" -> -2).
- `_estimate_n_gpu_layers` 的 `layer_size <= 0` 分支仅空文件时可达 (返回 -1 无害),
  属可留可删的防御代码, 不动.
- "chat handler 渲染消息时不改写 messages 入参" 的行为性假设 (image 逐张模式
  复用同一 messages 列表, 仅原位改写 image_url 项) 已对照当前 wheel 源码确认:
  MTMDChatHandler 基类注入默认 system 消息时以 `[{...}] + messages` 构造新列表,
  不动调用方入参. 该假设无法静态锁定, 升级 wheel 时仍按 AGENTS.md 对接面清单
  人工复核此处.
- ReasoningBudgetSampler (`reasoning_budget` 采样器) 只监听生成 token 流,
  看不到 prompt 内已存在的 reasoning_start: 模板预注入 `<think>` 的形态
  (如 Qwen3.5 GGUF 内嵌模板, `add_generation_prompt` 时尾部注入 `<think>\n`,
  enable_thinking 未定义即走该分支) 下采样器停在 IDLE 等不到开标签, 经
  `reasoning_start_max_tokens=32` 安全窗静默转 DONE, `reasoning_budget=0`
  完全失效 (思考照常生成). wheel 为此形态提供 `reasoning_start_in_prompt=True`:
  初始态改 COUNTING, budget 0 时立即 FORCING 在首个生成 token 处强制闭合.
  已对照 `_internals.py` 状态机源码与本地 Qwen3.5 GGUF 模板确认.
- 文本路径 (无 chat_handler) 无模板变量透传通道: `create_chat_completion`
  签名无 `**kwargs`, `chat_formatter_to_chat_completion_handler` 调 formatter
  时只传 messages/functions/tools 等固定参数; `Jinja2ChatFormatter.__call__`
  虽接受 `**kwargs` 合并进模板上下文, 但调用链没有出口把请求期参数送到那里,
  `enable_thinking` 这类模板级关思考开关 (LM Studio 的实现方式) 在该 wheel
  文本路径不可达, 关思考只能走 reasoning_budget 采样器机制.
- 文本路径 `llm.chat_format` 的可达值已对照 `llama.py` 构造逻辑确认: 插件不
  显式指定 chat_format 时, 只会是 guess 精确匹配的内置格式 (chatml /
  mistral-instruct / llama-3, 均不预注入 `<think>`), GGUF 内嵌模板的
  "chat_template.default", 或无模板时的 fallback "llama-2";
  `instruct.think_open_preinjected()` 只对 "chat_template.default" 渲染探测
  即覆盖全部预注入可能.
